# 네이버 클라우드 서버 구성 피드백 대응 정리

## 1. WAS 사양 관련
- **LLM 사용 방식**: 외부 OpenAI API 호출 구조이며, 자체 추론 모델을 서버에서 직접 구동하지 않습니다. 다만 Celery 워커가 프롬프트 전처리, 응답 후처리, 시나리오·주문 로직을 동시에 처리하므로 vCPU 8·RAM 32GB 단일 서버 수준을 기본으로 유지하려 합니다.
- **Redis 운영**: 현재는 Docker 컨테이너로 Redis를 포함한 Compose 구성을 계획하고 있습니다. 다만, 장애 복구와 운영 편의성을 고려해 NCP “Cloud DB for Redis” 도입 옵션도 검토하고, 필요 시 `mdfiles/serverfeedback.md`에 비교 결과를 추가하겠습니다.

## 2. Web 서버/로드밸런서 제안
- **스펙 조정**: 정적 자산 서버의 스펙이 높다는 지적은 타당합니다. 현재 문서에는 단일 서버 구성(49만 원대)과 2대 분리 대안(41만 원대)을 모두 기록해 두었고, 정적 자산 부하가 낮거나 CDN을 도입하면 더 낮은 사양으로 조정 가능합니다.
- **이중화 및 SSL 관리**: NCP 로드밸런서를 활용하면 이중화와 인증서 관리가 간편해지는 장점이 있어, 2대 분리 시나리오에 로드밸런서 사용을 포함시키는 방향을 검토 중입니다. 최종 결정은 `mdfiles/server.md`에 반영할 예정입니다.

## 3. Cloud DB for MySQL 고가용성
- **현재 계획**: 비용 제약으로 단일 노드 Cloud DB를 기본안으로 잡았지만, 고가용성 요구가 확정되면 Master/Standby 구성으로 전환하는 시나리오를 추가할 예정입니다.
- **아키텍처 정리**: `mdfiles/WhatcanIdo.md`와 신규 다이어그램에 단일 구성과 Master/Standby 구성을 모두 표현하고, 회의 결정에 따라 `mdfiles/server.md`에서 확정안을 관리하겠습니다.
